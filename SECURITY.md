# Security Policy

MC is a compiler and kernel research prototype. Do not treat the current `master`
branch or any `0.7.0-dev` snapshot as production hardened.

## Supported Versions

| Version | Security support |
|---|---|
| `master` / `0.7.0-dev` | Best-effort fixes only; no embargo or SLA. |
| Future tagged releases (`v*`) | Best-effort security fixes for the latest tagged release line until it is superseded. |

## Tagged Release Support Policy

No tagged release has shipped yet. When the first `v*` tag is published, that tag
becomes the only supported release line until a newer tag supersedes it. Security
fixes are expected to land on `master` first, then be included in the next tag or
backported to the latest tag when the fix is small, isolated, and does not require
an incompatible language/runtime change.

Older tags are retained for reproducibility but are unsupported unless a release
note explicitly says otherwise. A production claim must also name the exact tag,
artifact digest, and release inventory/SBOM generated by the release workflow.

## Reporting

Report suspected vulnerabilities privately through
[GitHub Private Vulnerability Reporting](https://github.com/Haofei/modern-c/security/advisories/new).
Do not open a public issue for an undisclosed vulnerability. Public issues remain
appropriate for ordinary correctness bugs that do not require coordinated disclosure.

The project targets acknowledgement within three business days and an initial
severity/affected-version assessment within seven business days. These are
best-effort targets rather than an SLA. Reporter and maintainer should agree on an
embargo and disclosure date after triage; fixes should be prepared privately when
premature disclosure would put users at risk. Confirmed vulnerabilities may receive
a GHSA and CVE when the affected surface has been distributed or the identifier is
useful for downstream tracking.

Useful report fields:

- host OS and architecture
- Zig, clang/LLVM, and QEMU versions when relevant
- the command that was run
- input files or a minimized reproducer
- expected outcome versus actual diagnostics, crash, hang, or emitted output

## Scope

Security-sensitive compiler issues include crashes or hangs on user input, silent
miscompiles, fail-open diagnostics, unsupported features accepted silently, and
incorrect emitted runtime checks. Kernel/appliance issues include violations of the
agent isolation, broker capability, resource accounting, audit, and hostile-input
boundaries described in `docs/threat-model.md` and `docs/security-review.md`.

Vendored third-party engines are part of the trusted computing base where linked.
Track upstream advisories for those projects until this repository has a formal CVE
intake and vendored-dependency update process.
