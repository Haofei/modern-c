| `[x]` | Production MIR optimizer use | Fact-gated optimizer (spec §E.4): `mir.buildOpt`/`verifyOpt` + `mcc verify\|lower-mir\|emit-c\|emit-llvm --optimize`, **off by default** (byte-identical standard pipeline). **Both backends consume the optimized MIR to elide the *emitted* runtime check** — each elided check's operand source point is recorded so the C and LLVM emitters skip exactly it. ## Standard library and MC-C2 profile

| Status | Item | Current code evidence | Next step |
|---|---|---|---|
| `[x]` | `std/mmio` register-field helpers and IO-memory copy | `std/mmio.mc` (spec §28.6) layers two things on the language's typed MMIO (§17) + `fence`/`cpu` intrinsics: **register bit-fields** — a `RegField` is a `(shift, width)` slice of a 32-bit register and `reg_field_get/_set` (+ single-bit `reg_bit_*`) extract/replace it with the mask built and bounds-checked in one place; the inserted value is masked to the field width (no bleed into a neighbour). They are pure `const fn`s, so a field built from constant `(shift, width)` folds and is verified at comptime. And the **IO-memory copy** — `mmio_write_block`/`mmio_read_block`/`mmio_write32`/`mmio_read32`/`mmio_modify_field` burst bytes/words between CPU memory and a device window with the fence on the correct edge (release before device writes, acquire after device reads); the raw load/store is the single `unsafe` site. Because typed MMIO is reached through a concrete `MmioPtr<Device>`, the helpers operate on the register value / a raw `PAddr` window rather than being generic over a device struct. Fixture `tests/qemu/mem/mmio_demo.mc` runs the field algebra, the block round-trip, and the ordered RMW; `mmio-test` runs it on **C and LLVM** (host-suite) and includes a comptime-fold check. | Optional: add `set_bits`/`clear_bits` sugar and a ported driver demo once a device needs it. |
| `[x]` | Library-scale DMA ownership protocols | `std/dma.mc` + `move` checking + the single-buffer DMA/cache fixtures are now joined by `tests/spec/dma_multidevice.mc`, which carries the §18.2 CPU-owned/device-owned typestate across **several engines (TX/RX/copy) and several in-flight buffers at once** — the library-scale case. It shows the linear discipline scales (ownership is single, not device-tagged): accepts two engines with a buffer in flight on each simultaneously, a buffer migrating between engines, a 3-slot ring drained in order, and conditional engine dispatch consumed on every path; rejects double-submitting one buffer to two engines (`E_USE_AFTER_MOVE`), submitting a CPU-owned handle where a device handle is required (`E_NO_IMPLICIT_CONVERSION`), leaking one engine's buffer while the other is cleaned up (`E_RESOURCE_LEAK`), and asymmetric branch dispatch (`E_MOVE_BRANCH_MISMATCH`). Validated by the inline-`EXPECT_ERROR` conformance test (`src/spec_tests.zig`, bidirectional) and the C + LLVM emit sweeps. The "decide where simulation belongs" question is answered: the typestate is a **sema/compile-time** property, so the state machine is checked statically in the spec sweep rather than run in QEMU. | Optional: a runnable host driver that executes a coherence simulation, if a future device needs runtime cache-maintenance behavior covered. |
| `[x]` | Generational handle opacity | `opaque struct` (spec §31) makes a struct's fields private to its associated functions (`impl Name`): outside code may hold/pass/return a value but cannot construct, read, or write it (`E_PRIVATE_FIELD`). Membership is the leading owner segment, so it survives generic monomorphization (`GenRef__u8` ↔ `GenRef__resolve__u8`) and textual-inclusion imports. `std/arena.mc` `GenRef<T>` and `std/pool.mc` `PoolRef<T>` are now `opaque`, minted/inspected only through their `impl` accessors, so a handle cannot be forged with a chosen generation/index. Verified: `tests/spec/opaque_field.mc` (plain + generic, construct/read/write) and the `pool-/genref-/wrap-/arena-/net-arena-test` runtime suites all pass. | Optionally extend the same opacity to other capability-bearing structs (page/DMA handles, grant tables). |
| `[x]` | Advanced packed ABI validation | `abi-test` (`tools/toolchain/abi-test.sh` + `tests/toolchain/abi_layout.mc`, wired into `m0`) is a three-layer golden test: MC's comptime layout model folds `sizeof`/`alignof`/`field_offset`/`bit_offset` asserts; the emitted C `_Static_assert`s the *same* sizes/alignments/offsets against clang's real `sizeof`/`_Alignof`/`offsetof` (host C ABI agreement); and it checks the C backend emits `volatile` MMIO registers with the correct `@offset` padding and that the LLVM backend agrees on the overlay byte-array and volatile load/store shape. Covers packed bits (u8/u16), overlay unions (mixed widths), a packed-bits field nested in a struct, an overlay nested in a struct, and an MMIO register block. | Optional: extend the layout model so a packed-bits type may itself be an overlay-union field (currently `UnsupportedCEmission`); add `[N]packed`/nested-array ABI cases. |
| `[x]` | Precise asm per compiler/architecture | The backends lower precise-asm operands with generic `"r"` constraints and keep the *named* registers only as provenance, so the compiler now **verifies** the register facts the contract previously only trusted (`src/sema.zig` `checkAsmConstraints`): `E_ASM_UNKNOWN_REGISTER` (not a register on any supported ISA), `E_ASM_ARCH_MIXED` (one block names registers of more than one ISA — x86-64 `rax` with RISC-V `a0`), `E_ASM_REGISTER_CONFLICT` (a register bound to two operands), `E_ASM_CLOBBER_CONFLICT` (a clobber that also holds an operand).
